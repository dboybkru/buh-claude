# AI assistant — ручной smoke-checklist (Sprint 6A + 6B + 6.1)

Цель: проверить безопасность и UX AI-помощника глазами, без обращения к внешним моделям. Используется Mock-провайдер (детерминированный, без сети).

## Подготовка

Один раз:

```bash
docker compose up -d postgres
cd backend && npm run seed     # тестовый пользователь test@buhclaude.local / superpass1
npm run dev                    # backend:3001 + frontend:5173 параллельно
```

Залогиниться под `test@buhclaude.local` / `superpass1` на http://localhost:5173.

---

## Часть 1. Настройки провайдера

1. Открыть **`/ai/settings`**.
2. В выпадающем списке «Провайдер» выбрать **Mock (dev/test)**. baseUrl автоматически становится `mock://local`, поле API Key прячется (mock не требует ключа).
3. Нажать **Сохранить** → toast «Настройки AI сохранены». Бейдж в правом верхнем углу должен стать **«Настроен»**.
4. Нажать **«Проверить подключение»** → справа от кнопки появится «✓ ok» (или близкое сообщение).
5. (опционально) Нажать **«Получить модели»** — должен прийти список из трёх mock-моделей.

**Ожидание UX:**
- После сохранения поле API Key очищено; повторно вводить ключ не нужно (mock-провайдер).
- `maskedApiKey` в API-ответе == `(mock)` для Mock-провайдера.
- Под полем подсказка «AI не выполняет действия без вашего подтверждения».

## Часть 2. Позитивный сценарий — create_counterparty

6. Открыть **`/ai`**.
7. В верхней карточке «Контекст» выбрать организацию (из seed по умолчанию — ООО «Альфа»).
8. В поле сообщения отправить: **«Создай контрагента ООО Ромашка ИНН 7701234567»**.

**Что должно появиться:**
- Карточка «AI предлагает план».
- Заголовок: «Создать контрагента ООО Ромашка (ИНН 7701234567).»
- Бейдж confidence ≈ 0.92 (высокий).
- Бейдж «Создаст данные после подтверждения».
- Блок ActionPreview с типом «Создание контрагента» и реквизитами.
- Кнопка **«Подтвердить действия»** активна, кнопка «Отклонить» рядом.

9. Нажать **«Подтвердить действия»**.

**Что должно появиться:**
- Toast «Применено: 1».
- ConfirmResultCard: applied = 1, errors = [].
- Ссылка «Открыть контрагента» → ведёт в `/counterparties` с фокусом на новой записи.

10. В **Справочники → Контрагенты** должен появиться «ООО Ромашка» с ИНН 7701234567.

- ActionPreview «Создание счёта» с позицией «консультацию», qty=1, price=10000, vatRate=без НДС.

**Что должно появиться:**
- Toast «Применено: 1».
- В **Документы → Счета** появится новый счёт с номером вида `СЧ-NNNN/2026` (auto-number), статус DRAFT, контрагент = «ООО Ромашка», сумма 10000 ₽, НДС 0.

## Часть 6.5. Платежи и распределение (Sprint 6C)

### create_payment IN с allocations

22a. Убедиться, что есть неоплаченный счёт (из части 3) и контрагент в контексте.
22b. На `/ai` отправить: **«Создай входящий платёж на 10000 по счёту»**.

**Что должно появиться:**
- ActionPreview «Создание платежа» с бейджем «входящий».
- Сумма 10000 ₽, направление IN.
- Блок «Распределение» с привязкой к invoiceId.
- Если сумма ≤ остатка → нет аванса; если больше → блок «Аванс».

22c. Подтвердить.

**Что должно появиться:**
- ConfirmResultCard: applied=1, ссылка «Открыть платёж» → `/payments`.
- В **Финансы → Платежи** появится платёж 10000 ₽, IN, BANK.
- Связанный счёт перешёл в **PAID** или **PARTIALLY_PAID** в зависимости от суммы.

### create_payment OUT

22d. Отправить: **«Создай исходящий платёж поставщику 25000»**.

**Что должно появиться:**
- ActionPreview «Создание платежа», бейдж «исходящий».
- Подсказка: «Исходящий платёж — без привязки к нашим счетам».
- НЕТ блока allocations.

22e. Подтвердить → платёж OUT появится в **Финансы → Платежи**.

### suggest_payment_allocations (read-only)

22f. Отправить: **«Распредели платёж 50000 по счетам контрагента»**.

**Что должно появиться:**
- ActionPreview «Предложение распределения», бейдж «не изменяет данные».
- Сумма 50000 ₽.

22g. Подтвердить.

**Что должно появиться:**
- ConfirmResultCard с блоком **«Предложение распределения на YYYY-MM-DD»**.
- Поля «Сумма платежа / Распределено / Аванс».
- Список счетов с FIFO-разнесением, у каждого `reason` («Просрочен с …» или «Старейший непогашенный счёт»).
- Если сумма больше долга → ненулевой `advanceAmount` + warning.

22h. **Проверить, что платёж/распределение в БД НЕ созданы**:
- **Финансы → Платежи** — никаких новых записей.
- Сумма по неоплаченным счетам не уменьшилась.

## Часть 8. Негативные сценарии

### 8a. missingFields блокирует Confirm

23. Отправить: **«Создай контрагента ООО Без ИНН»** (без ИНН).

**Ожидание:**
- В action plan: `missingFields: ["inn"]`, `actions: []`.
- Кнопка **«Подтвердить действия» disabled**.
- В блоке missing указано `inn`.

### 8b. Repeat confirm отклоняется

24. Создать любой action plan (например счёт), подтвердить.
25. Через DevTools/Network взять `actionPlanId` и повторно отправить POST `/api/v1/ai/action-plans/<id>/confirm`.

**Ожидание:** HTTP 409 «ActionPlan уже в статусе CONFIRMED — повторное применение запрещено».

### 8c. Дубль акта по счёту

26. Отправить «Создай акт по последнему счёту» дважды подряд — между запросами не создавая новый счёт.

**Ожидание:**
- Первый раз — applied=1.
- Второй раз — applied=[], errors[0].error содержит «уже создан акт».

### 8d. Cancelled-счёт отклоняется

27. В **Документы → Счета** взять любой счёт, перевести его статус в **CANCELLED**.
28. На `/ai` отправить «Создай акт по последнему счёту» (mock возьмёт первый счёт из recentInvoices — это cancelled).
29. Confirm.

**Ожидание:** errors[0].error содержит «отменённого счёта».

### 8e. Cross-organization защита

30. (Требуется два пользователя или manual curl-запрос с поддельным organizationId.) Создать ActionPlan под user A с payload, ссылающимся на organizationId user B.
31. Confirm → executor должен отклонить с «Организация не найдена или принадлежит другому пользователю».

### 8f. Неизвестный approvedAction id

32. Создать plan с одним action `a1`. В confirm передать `{"approvedActions": ["fake-id"]}`.

**Ожидание:** result.applied=[], skipped содержит `a1` (он не в approved), но также errors включает «Неизвестные approvedActions: fake-id» — или skipped с reason описывает это явно (зависит от реализации).

### 8g. Expired plan

33. Подождать 24 часа (TTL) ИЛИ напрямую через DB установить `expiresAt < now()` у DRAFT plan.
34. Confirm → 409 «Срок действия ActionPlan истёк», plan переводится в EXPIRED.

## Известные ограничения (что НЕ должно работать) — состояние после Sprint 6C

- AI **создаёт платежи** через `create_payment` action (Sprint 6C) **только после явного подтверждения**.
- AI **распределяет платежи** только через явные allocations в `create_payment` ИЛИ через read-only `suggest_payment_allocations` (показывает, но не пишет).
- AI **НЕ импортирует** банковскую выписку — bank-import работает только вручную через `/bank-import`. AI bank-import не входит в Sprint 6C.
- AI **НЕ редактирует** и **НЕ удаляет** существующие документы / контрагентов / договоры / платежи — нет `update_*` / `delete_*` action.
- AI **НЕ меняет** `AiSettings` или `AiAuditLog` — это системные ресурсы.

Если в одном из этих сценариев AI вернёт action — это бага: SYSTEM_PROMPT и whitelist `ALLOWED_ACTION_TYPES` должны это блокировать.
